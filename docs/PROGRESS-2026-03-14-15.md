# Agent Runtime — Progress Report: March 14-15, 2026

## 1. Executive Summary

In 48 hours, the agent runtime went from a working prototype (v0.2) to a deployment-ready system (v0.3.7). Eight sub-versions were shipped. The work spanned three distinct phases: long-term stability hardening (v0.3.0-v0.3.1), cognitive quality improvements driven by soak testing (v0.3.2-v0.3.6), and deployment readiness (v0.3.7).

The headline results:
- Object hallucinations went from "rampant" to **zero** across 4 consecutive tests
- Valence flatline (stuck at 0.000 for 70% of ticks) was diagnosed and eliminated
- Speech quality transformed from robotic metric-parroting to natural experiential language
- The 8-hour overnight soak test ran 1,948 ticks with 31 sleep cycles and zero crashes
- The model was upgraded from llama-3.1-8b-instant to llama-3.3-70b-versatile, doubling the agent's interact rate from 12% to 30%
- Every stability issue in the known backlog was fixed except one (process watchdog)

The pattern that emerged across all fixes: **sensation, not instruction.** When the agent was told what to do via prompts, it often ignored the guidance. When its perception was shaped so that undesirable inputs simply did not exist, the problems vanished. This principle — hard constraints over soft prompts — became the project's central engineering thesis.

## 2. Starting Point: v0.2

Version 0.2, completed around March 10, introduced the five-stage cognitive loop (SENSE, FEEL, THINK, ACT, REFLECT), replacing the simpler OBSERVE, THINK, ACT cycle of v0.1. It added:

- **InternalState** — Two-axis affect model (valence/arousal) updated by environment signals, action results, novelty, and social events
- **DeltaDetector** — Diffed observations between ticks and narrated changes in natural language
- **RepetitionGuard** — Tracked recent actions and warned the LLM when patterns repeated
- **Adaptive heartbeat** — Tick interval ranged from 4s (high arousal) to 15s (calm), controlled by arousal level
- **Sleep cycle with self-reflection** — The agent could evolve its own personality during consolidation

v0.2 proved that the cognitive loop worked in controlled test scenarios (107 ticks across 10 scenarios). But it had never been tested for more than a few minutes at a time. The project's target — an agent running continuously for an entire summer as a thesis installation — demanded a fundamentally different level of reliability.

### Known problems at the start of March 14

1. **DailyLog was a disk killer.** Every 8-second tick read the entire log file, appended a line, and rewrote it. That was 21,600 read+rewrite cycles per day. On a Raspberry Pi's SD card, this was a path to hardware failure within weeks.

2. **Sleep was the only garbage collection.** Memory consolidation, log cleanup, and state reset all ran exclusively during the sleep cycle. If the agent crashed before sleeping, everything grew unbounded.

3. **Context window overflow.** Sleep consolidation passed the full day's log (~400KB, ~100K tokens) to an 8K-token model. The output was silently garbled. Memories formed during sleep were garbage.

4. **No crash recovery.** On restart, internal state (valence, arousal) reset to zero. The agent had no emotional continuity across crashes.

5. **Persona drift was unbounded.** The agent evolved its personality during sleep with only a soft prompt ("be subtle") as a guard. Over months, the persona would become unrecognizable.

6. **Object hallucinations.** tools.md was append-only. Objects written to memory persisted even after disappearing from the world. The agent kept referencing a "terminal-01" that no longer existed.

7. **No process monitoring, no rate limiting, no backoff on errors.**

## 3. Version Progression

### v0.3.0 — Long-Term Stability Overhaul (March 14, early)

**Problem:** The runtime would not survive more than a few days of continuous operation. At least 6 systems accumulated unbounded state, and disk I/O patterns would degrade the Pi's SD card.

**Key decisions and reasoning:**

- **DailyLog switched to in-memory buffer with batch flush.** Instead of reading and rewriting the file on every tick, entries are held in RAM and flushed periodically via `appendFile`. Disk I/O dropped from 21,600 writes/day to approximately 288. The decision was to accept a small window of potential data loss (entries in the buffer at crash time) in exchange for a 99% reduction in disk writes. For a thesis installation on SD card hardware, this was a clear tradeoff.

- **Background maintenance timer added.** A fallback GC timer runs hourly, independent of the sleep cycle. If the agent has not slept in 24 hours (crash loop, configuration error, etc.), maintenance runs anyway. The principle: critical processes must not depend on a single codepath.

- **Sleep consolidation capped at 200 lines.** Rather than passing the entire daily log (~100K tokens) to an 8K-token model, the consolidation input was limited to the 200 most recent lines plus salient events. This was a pragmatic cap — the LLM can only process what fits in its context window, and passing more produced garbled output.

- **Crash recovery via state checkpoints.** Internal state (valence, arousal) now checkpoints to `state-checkpoint.json` every 5 minutes. On restart, the last checkpoint is loaded if it is less than 1 hour old. This ensures emotional continuity across crashes without burdening every tick with a disk write.

- **Persona drift guard with quantitative measurement.** A `_measureDrift()` function compares the current persona against the original across array fields (traits, values, fears, quirks). If more than 60% of original items have been replaced, evolution is blocked. The threshold was chosen as a balance: enough room for genuine growth, tight enough to prevent identity loss over 180+ sleep cycles per month.

- **Token budget for prompts.** The prompt builder now estimates token count and truncates memory.md if the total exceeds ~7,800 tokens. This prevents context window overflow during normal operation.

- **tools.md hash-based write skipping.** A djb2 hash of the file content prevents redundant writes when nothing has changed.

- **WebSocket exponential backoff.** Reconnection delay starts at 5 seconds and caps at 5 minutes, preventing connection storms.

- **SSE stale client cleanup.** Dead SSE connections are cleaned up after 5 minutes of inactivity.

- **/metrics endpoint.** Exposes runtime metrics for external monitoring.

**Validation:** Unit tests passed. No soak test yet — this was infrastructure hardening.

### v0.3.0 — First Soak Test (March 14, mid-morning)

The first long-duration soak test was run: 2 hours, 514 ticks, 10 environmental phases, 3 sleep/wake cycles.

**What it validated:**
- Sleep cycles fired exactly on schedule (30 min active, 5 min sleep). All 6 consolidation passes completed successfully.
- Memory grew from 11 to 22 entries. Skills from 0 to 10. The agent was learning.
- Emotional tracking worked: stress phases hit valence -0.262, flourishing peaked at +0.345.
- Adaptive heartbeat responded correctly: 8.7s during stress, 15s during calm.
- Synth mode produced zero spatial action leaks across 43 ticks.

**What it revealed:**
- **Memory duplicates.** The 8B model's consolidation pass could not deduplicate. After 2 hours, memory.md contained 8 near-identical entries about "adding new chords can break the repetition." The LLM prompt said "merge redundant entries" but the model just kept everything.

- **Speech was repetitive.** "Interesting" and "Right" appeared over and over.

The soak test ran on v0.2 code, meaning the v0.3.0 stability fixes had not yet been tested under load. But it established the testing methodology that would drive every subsequent version.

### v0.3.1 — The 3-Month Readiness Audit (March 14, afternoon)

**Problem:** A systematic audit of every source file for a 3-month continuous installation revealed 20 issues. Seven were classified as no-ship blockers.

**The seven critical fixes:**

1. **Memory corruption protection.** Sleep consolidation passed the entire memory.md to the LLM and wrote back whatever it returned. The only guard was `result.length > 10`. If the LLM returned garbled markdown, memory was permanently destroyed with no backup and no rollback. Over 3 months with 6 sleep cycles/day, that was 540 chances to corrupt memory.

   *Fix:* backup, validate, write. If the output lacks markdown headers and list entries, restore from backup. Every consolidation write is now atomic.

2. **Destructive tools.md refresh removed.** During sleep, `_refreshTools()` asked the LLM to "clean up" tools.md. But tools.md is rebuilt from the live observation every tick. The LLM would reformat the header, breaking the regex match, causing duplicate sections to accumulate forever.

   *Decision:* Delete the entire method. If a file is rebuilt from ground truth every tick, there is no reason for the LLM to touch it during sleep. This was a case where a "nice to have" feature was actively destructive.

3. **Persona evolution type validation.** The LLM could return `{"traits": "bold"}` instead of `{"traits": ["bold"]}`. The persona file would be written. Next tick: `persona.traits.join(', ')` crashes. Every tick. Unrecoverable without manual editing.

   *Fix:* Type-check every field before merging. Arrays must be arrays. Objects must be objects. Persona file backed up before any write.

4. **Immutable persona baseline.** The drift guard compared current personality against `_originalPersona`, which was loaded on the first sleep cycle, not on boot. After a crash-restart, the drifted persona became the "original." After 10 restarts over a summer, the 60% drift threshold was meaningless — the baseline kept shifting.

   *Fix:* Save an immutable `persona-baseline.json` on first-ever boot. Load from that file on every subsequent startup. The original is the original, forever.

5. **Skills extraction hallucination guard.** The LLM was asked to "find procedural knowledge the agent learned." The agent does not learn procedures — it does `move_to`, `interact`, `speak`. But the LLM invented an entire skill tree ("Territory Management: Claiming Territory — Move to a new location and assert dominance"). This fabricated knowledge was fed back into every tick's prompt.

   *Fix:* Constrained the prompt to extract ONLY from the activity log with strict rules. Added validation and backup.

6. **Memory truncation order fixed.** When the prompt exceeded the token budget, `memory.slice(0, memory.length - overBy)` cut from the end. The memory.md sections are ordered: Relationships, Learned Facts, Important Memories. So the most valuable section — Important Memories — was cut first.

   *Fix:* Truncation now specifically targets the middle of Learned Facts (the largest, least critical section). Important Memories and Relationships are protected.

7. **Memory entry length cap.** Nothing stopped the LLM from writing a 500-character paragraph as a "learned fact." Over months, individual entries would bloat and blow the token budget, triggering truncation that cut other memories.

   *Fix:* Hard cap at 120 characters per memory entry, enforced both in the decision handler and in the append method.

**Design principle established:** Every LLM output must be validated before it is persisted. Every persistent file must have a backup. The difference between a demo and a thesis is that writes are guarded at every layer.

### v0.3.2 — Anti-Repetition Overhaul (March 14, late afternoon)

**Problem:** The v0.3.1 soak test (45 minutes, ~340 ticks, 13 phases) revealed three issues: 3 remaining hallucinations, monotone emotional descriptions ("Feeling steady" covered 70% of ticks), and repetitive speech.

**Six fixes shipped:**

1. **Disappeared-object tracking.** Track recently-vanished objects for 10 ticks after removal. Inject an explicit GONE warning into the prompt: "The following objects have DISAPPEARED and are NO LONGER HERE: pillar-01. Do NOT mention, interact with, or speak about them."

   *Reasoning:* The agent sees objects disappear from the Nearby Objects section, but nothing explicitly says "it is gone." An 8B model needs the absence to be stated, not inferred.

2. **Fuzzy speech dedup.** Extract keywords from each utterance (stop-word removal), compare keyword sets between recent speech. 60%+ keyword overlap triggers a warning. The same approach used for memory dedup — if keyword overlap works for facts, it works for speech.

3. **Emotional descriptions expanded from 9 to 16.** The neutral catch-all covered valence -0.1 to 0.15 and arousal -0.3 to 0.5 — a huge range. The neutral band was narrowed to v: -0.05 to 0.05, a: -0.15 to 0.15. Mid-range states added: "comfortable focus," "awake and aware," "flat and disengaged."

   *Reasoning:* The agent spends most of its time in the middle of the emotional range. That is exactly where it needs the most resolution.

4. **Working memory merged events.** Each tick generated 2 working memory events (action + result). With a 12-slot buffer, the agent could only see 6 ticks of context. Action and result events now merge into one slot. Buffer increased from 12 to 20.

   *Impact:* The agent now remembers 20 ticks of context instead of 6.

5. **Object position narration.** Objects now include distance or coordinates. "pillar-01 3.2 away" instead of just "pillar-01."

6. **Version bump to v0.3.2.**

### v0.3.3 — Valence Flatline Fix (March 14, evening)

**Problem:** The second soak test on v0.3.2 showed hallucinations dropping from 3 to 1. But valence flatlined at exactly 0.000 from tick 52 onwards for the entire rest of the test.

**Root cause:** Successful actions had NO effect on valence. Only failures mattered (-0.15). With no environment signals in half the test phases, valence decayed to zero and stayed there. The agent was successfully exploring, interacting, discovering — and feeling nothing.

**The fix: asymmetric reward.**

```js
if (!context.actionResult.success) {
    this._nudgeValence(-0.15)     // failure: sharp negative
} else {
    this._nudgeValence(0.02)      // success: mild positive
    if (context.actionResult.action === 'interact') {
        this._nudgeValence(0.04)  // exploration: rewarding
    }
}
```

Success generates mild positive valence. Failure generates sharp negative valence. This is negativity bias — it mirrors how biological systems work. The penalty is 7.5x the reward. The agent does not need the environment to tell it to feel good; doing things well is its own reward.

**Additional fixes:**
- GONE window extended from 10 to 30 ticks (~4 minutes) to catch the one remaining hallucination (tick 138, 46 ticks after object removal)
- Soak test phases updated with baseline signals (previously, half sent null vitality/warmth, making the test unrealistically harsh)

### v0.3.4 — 8-Hour Overnight Soak Test (March 15, morning)

**Problem:** The system needed validation at scale. Previous tests were 45 minutes. The target installation runs for months.

**The overnight test:** 8 hours, 1,948 ticks, 5 complete 97-minute phase cycles, 31 sleep/wake transitions.

**Results:**
- Zero crashes, zero WebSocket disconnects, zero unrecoverable errors
- Memory started at 15 entries, consolidated down to 10. Pruning worked.
- Valence ranged from -0.22 to +0.40 (the flatline was eliminated)
- 7+ distinct emotional descriptions appeared across all 5 cycles
- Empty world phase produced zero hallucinations across all 5 cycles

**New insights:**

- **Memory vs. hallucination distinction.** The soak test reported 16 hallucinations. Investigation revealed 6 were false positives (the detector used `.includes("pond")` which matched "responds" and "responding" as substrings). Of the 10 real hits, approximately 5 were legitimate reminiscing — the agent reflecting on past experiences, not claiming absent objects were present.

   This distinction matters. "I remember the pond" is not a hallucination. "The pond is interesting" when the pond is gone — that is a hallucination. The prompt was updated with two separate rules: (1) only interact with objects listed under Nearby Objects RIGHT NOW, and (2) you may remember past experiences, but always use past tense for things no longer present.

   *Decision:* We want the agent to build a sense of history. Total silence about absent objects would make the agent amnesic. The goal is to prevent hallucinated presence, not hallucinated memory.

- **Soak test false positive fix.** `.includes("pond")` replaced with word boundary regex `\bpond\b`.

- **Model research.** Evaluated qwen3:4b as a local fallback — generational improvement over qwen2.5:3b, better hallucination control, fits in ~3GB RAM.

### v0.3.5 — Model Upgrade + Signal Translation (March 15, midday)

**Problem:** Two separate issues converged. The 8B model's action distribution was heavily biased toward `move_to` (53%, with interact at only 12%) — the agent wandered more than it engaged. And the agent kept parroting raw metric names ("vitality," "resonance") in its speech because those literal words appeared in the observation.

**Key decisions:**

1. **Cloud model upgrade: llama-3.1-8b-instant to llama-3.3-70b-versatile.** Both run on Groq's free tier. The 70B model is dramatically more capable at following complex instructions, understanding context, and generating varied behavior. The tradeoff is higher latency per request (acceptable given the 4-15s tick interval) and lower rate limits (addressed in v0.3.7).

2. **Local fallback upgrade: qwen2.5:3b to qwen3:4b.** A generational leap — better hallucination control, native tool calling, matches Qwen2.5-7B quality at half the size.

3. **Signal descriptions: raw metrics replaced with felt experience.** This was the single most impactful design decision of the two days.

   Before: `Environment: vitality: 0.55, resonance: 0.20, warmth: 0.50`
   After: `Environment: There is a healthy energy here, things feel vibrant. The atmosphere is quiet and still. The air feels neutral — neither warm nor cold.`

   The agent never sees the word "vitality." It experiences what vitality feels like. The same principle already applied to internal state — this extended it to environment perception. When you show data to an LLM, it talks about data. When you show sensation, it talks about experience.

4. **Internal state numbers removed from prompt.** The agent sees "A comfortable focus — present and attentive" not "valence: 0.22, arousal: 0.32."

5. **Speech rules updated.** "Say something SHORT, FRESH, and in your own voice — react to what you feel and see, don't analyze or explain."

**First 70B soak test results (45 minutes, 170 ticks):**

| Metric | 8B Model | 70B Model |
|--------|----------|-----------|
| move_to | 53% | 42% |
| interact | 12% | **30%** |
| speak | 21% | 21% |
| wait | 5% | 6.5% |
| hallucinations | ~1-3 | 4 (all false positives) |

The agent went from "mostly wandering" to "actively engaging with its world." The 70B model understood "be curious, explore, interact" in a way the 8B could not.

**Second v0.3.5 soak test (45 minutes, 167 ticks):**
- **Zero hallucinations.** First completely clean test in the project's history.
- Speech quality transformed completely — no mention of "vitality" or "resonance" in any speech. Examples: "Something feels off about this place," "The chill in the air is unsettling," "The emptiness is suffocating."
- Interact held steady at ~30%.

### v0.3.6 — Teaching Creativity Through Sensation (March 15, afternoon)

**Problem:** Speech was natural now, but the agent still occasionally repeated itself. Two patterns survived the existing filters: "Vibrancy is palpable here" appeared twice, and "Something feels off" appeared in two similar constructions across 167 ticks.

**The design question:** The current approach detected repetition and warned the LLM. This was reactive — a prompt-level fix, the weakest tool available. The agent had no reason to be creative; no consequence for repeating, no reward for novelty.

**The solution: creativity as felt experience.**

After every speech action, the system scores its creativity: keyword extraction, overlap comparison against the last 10 utterances. Score 0.0 = exact repeat. Score 1.0 = completely novel.

The score is fed back through valence — the only channel the agent trusts:

```js
applySpeechCreativity(score) {
    if (score < 0.4) {
        this._nudgeValence(-0.08)  // repetitive — world feels duller
    } else if (score > 0.8) {
        this._nudgeValence(0.03)   // creative — mild reward
    }
}
```

The agent never sees the score. It never knows why the world feels different. It just notices that when it says the same thing twice, everything gets a little worse. The penalty (-0.08) is stronger than the reward (+0.03) — the same asymmetric negativity bias used for action results.

**Why this matters architecturally:** This was the first time the emotional system was used not just to reflect the environment, but to shape behavior. The same principle could apply to spatial exploration (reward novelty, punish pacing), social interaction (reward engagement, punish ignoring), and memory formation (reward learning, punish rumination). All through sensation. All invisible to the agent.

### v0.3.7 — Deployment Readiness (March 15, evening)

**Problem:** The cognitive pipeline was complete. The remaining question was operational: would the system survive weeks without human intervention? The known issues backlog from the v0.3.1 audit still had 8 open stability items.

**Eight fixes shipped in one session:**

1. **429 rate limit handling.** Groq's free tier has rate limits. At 15 requests/minute during high arousal, the quota could be exhausted in 4 minutes. Previous behavior: generic error, fall through to Ollama, Ollama also fails if not running, agent goes silent.

   *Fix:* Detect 429 specifically. Enter 60-second cooldown. Auto-route to local Ollama during cooldown. When cooldown expires, cloud resumes automatically. The agent keeps thinking, just slower.

2. **Ollama periodic re-check.** `ollamaAvailable` was set once at startup. If Ollama restarted later, it was never rediscovered.

   *Fix:* Re-check every 5 minutes if initially unavailable.

3. **Promise leak on WebSocket disconnect.** Pending observe/action promises were never rejected on close. The tick hung for 5 seconds waiting for a timeout.

   *Fix:* Reject pending promises immediately in the close handler. 5 seconds of dead time per disconnect reduced to 0.

4. **Day boundary buffer.** Entries buffered at 23:59 flushed at 00:01 went into the wrong day's file.

   *Fix:* Tag each buffer entry with its target filename at creation time. Entries flush to the correct day regardless of when the flush runs.

5. **Tick counter persistence.** `tickCount` reset to 0 on restart, breaking temporal context. The agent thought it just woke up after running for hours.

   *Fix:* tickCount persists in the state checkpoint alongside valence/arousal. Restored on startup.

6. **Sleep consolidation rate limiting.** The 4 LLM calls during sleep used the same cloud API path as ticks. They could exhaust the quota and fail the first waking ticks.

   *Fix:* 5-second delay between sleep LLM calls to spread load.

7. **Prompt caching.** System prompt reads memory.md, skills.md, and tools.md every tick. At 8-second intervals: 10,800 file reads per day for content that changes maybe 10 times.

   *Fix:* Read cache in MemoryFiles with write-through invalidation. Writes update or invalidate the cache. File reads dropped from 10,800/day to approximately 30.

8. **DeltaDetector property mutation tracking.** Previously only detected objects appearing or disappearing by ID. Property changes (e.g., `interactive` toggling) were invisible.

   *Fix:* Now tracks property mutations on existing objects.

**Remaining backlog:** One item — process watchdog (systemd service for auto-restart on hang). Everything else is fixed.

## 4. Key Design Principles

Five recurring themes guided every decision across these 48 hours. They were not planned in advance — they emerged from repeatedly encountering the same class of problem.

### "Sensation, not instruction"

This is the project's central thesis, inherited from the OHMAR lineage ("situated, not sentient"). The agent should never be told what to do. Instead, its perception should be shaped so that desired behavior emerges naturally.

Applications across the two days:
- **Signal parroting** — The agent kept saying "vitality" and "resonance." The fix was not a prompt rule ("don't say vitality"). The fix was removing the word from the agent's perception entirely. It experiences what vitality feels like ("this place feels alive"), not the metric itself.
- **Emotional flatline** — The agent felt nothing during successful exploration. The fix was not a prompt ("act like you enjoy exploring"). The fix was wiring successful actions to mild positive valence. The agent feels good because doing well is its own reward.
- **Speech repetition** — The agent repeated itself. The fix was not a warning ("don't repeat yourself"). The fix was making repetition feel bad through negative valence, and creativity feel good through positive valence. The agent avoids repetition because repetition literally makes its world worse.

### "Hard constraints over soft prompts"

LLMs ignore prompt-level instructions with disturbing reliability. In the original v0.2 test, "only use available actions" was ignored 3 times in 107 ticks. Every fix that relied on a prompt instruction eventually failed. Every fix that enforced constraints in code worked immediately and permanently.

Applications:
- **Action validation** — Hard check in code, not a prompt rule
- **Memory dedup** — Keyword-based fuzzy matching in code before the LLM sees the data
- **Object hallucination** — Ground truth labels, GONE warnings, and disappeared-object tracking in code
- **Memory entry length** — Hard cap at 120 characters, enforced at two layers
- **Persona evolution** — Type validation, drift measurement, and backup/restore in code

### Asymmetric negativity bias

Penalties are always stronger than rewards. This mirrors biological systems and prevents two failure modes: positive feedback loops (where the agent gets stuck in a self-reinforcing positive state) and passivity (where the agent has no motivation to avoid mistakes because mistakes cost nothing).

- Action results: failure -0.15, success +0.02 (7.5:1 ratio)
- Interaction bonus: +0.04 on top of the success reward
- Speech creativity: repetition -0.08, novelty +0.03 (2.7:1 ratio)

The asymmetry ensures that the agent naturally recovers to a mild positive baseline through successful action, but feels real consequences for failure and repetition. The absence of punishment is itself a form of reward.

### Write-through caching and immutable baselines

Two data architecture patterns appeared repeatedly:

**Write-through caching** — MemoryFiles reads are cached in memory. Every write (append, consolidation, backup restore) updates or invalidates the cache. This eliminated 10,800 file reads per day. The principle: cache aggressively, but never let the cache become stale. Every write path must touch the cache.

**Immutable baselines** — The persona drift guard failed because the "original" persona was loaded at first sleep, not first boot. After crash-restarts, the baseline shifted. The fix: `persona-baseline.json` is written once, on first-ever boot, and never modified. The original is the original, forever. This pattern applies anywhere a reference point must survive restarts: the persona baseline, the original trait set, the initial tick counter.

### Validate everything the LLM produces

Every LLM output that gets persisted to disk must pass validation before being written. Every persistent file must have a backup taken before the write. This principle was established in v0.3.1 after cataloguing the failure modes:

- Memory consolidation could corrupt memory.md (fix: backup + markdown validation)
- Persona evolution could break the persona JSON (fix: type-check every field)
- Skills extraction could hallucinate entire skill trees (fix: constrain to log evidence)
- tools.md refresh could destroy section headers (fix: delete the method entirely)

The rule is simple: the LLM is a creative partner for generating decisions, but it is an unreliable partner for maintaining data integrity. Every write is guarded.

## 6. Model Upgrade Journey: 8B to 70B

The model upgrade was not planned at the start of the two days. It emerged from observing the 8B model's limitations across multiple soak tests.

### 8B model limitations (llama-3.1-8b-instant)

- **Action distribution.** 53% `move_to`, 12% `interact`. The agent wandered far more than it engaged. Complex prompt instructions about curiosity and exploration were partially understood but not reliably followed.
- **Memory consolidation.** Could not deduplicate. Eight near-identical entries persisted despite explicit "merge redundant entries" instructions. Required code-level fuzzy dedup as a pre-processing step.
- **Hallucination control.** Required multiple layers of defense (ground truth labels, GONE warnings, disappeared-object tracking) to prevent references to absent objects.
- **Speech quality.** Robotic and analytical. Parroted raw metric names. Responded to "don't analyze" prompts inconsistently.

### Why 70B was feasible

Both models run on Groq's free cloud API tier. The 8B-to-70B switch had no cost impact. The tradeoffs were:
- **Higher latency per request** — acceptable given the 4-15 second tick interval
- **Lower rate limits** — addressed by the 429 cooldown and Ollama fallback in v0.3.7
- **Better instruction following** — dramatically better across every metric

### 70B model improvements (llama-3.3-70b-versatile)

- **Action distribution.** 42% `move_to`, 30% `interact`. The agent actively engages with its world instead of wandering.
- **Hallucination control.** Zero hallucinations in the cleanest test. The memory-vs-hallucination distinction (reminiscing allowed, presence-hallucination blocked) was understood and followed correctly.
- **Speech quality.** Natural, experiential, varied. Combined with signal translation, the agent stopped being a research analyst and started being a being.
- **Instruction following.** Complex multi-part prompt rules were reliably followed. The agent understood "be curious" as a behavioral directive, not just text to echo.

### Local fallback upgrade

The local Ollama fallback was upgraded from qwen2.5:3b to qwen3:4b — a generational improvement. Key advantages: better hallucination control (~15% reduction), native tool calling, matches Qwen2.5-7B quality at half the size, fits in ~3GB RAM on the Pi's 8GB.

## 7. Remaining Work

### Immediate (pre-deployment)

- **Process watchdog.** The single remaining backlog item. A systemd service or external health check that restarts the Node.js process if it hangs. Without this, a hang requires manual SSH intervention.

### Near-term (deployment phase)

- **Cost optimization.** The 70B model on Groq's free tier has stricter rate limits than the 8B. The 429 cooldown and Ollama fallback handle this, but a paid tier or alternative provider may be needed for sustained high-arousal periods.
- **Multi-week soak test.** The 8-hour test validated the system at scale. A 1-week test on the actual Pi hardware would validate SD card wear, memory growth patterns, persona drift over many sleep cycles, and thermal behavior.
- **Signal translation tuning.** The current experiential descriptions are a first pass. With real-world testing, the vocabulary may need adjustment to produce more varied and contextually appropriate language.

### Medium-term (thesis goals)

- **Multi-agent deployment.** The architecture supports multiple agents, but the Pi can only run one at a time. Multi-agent would require either multiple Pis or a server-side deployment.
- **Environment integration.** The 3D world (anon-ai-world) is not yet connected. The sim-server needs a WebSocket agent endpoint. The agent runtime is environment-agnostic and ready.
- **Memory hierarchy.** The 50-entry memory.md cap means the agent forgets its first month by month three. A tiered system (core memories rarely pruned, knowledge updated when corrected, recent aggressively pruned) would give the agent genuine long-term identity.
- **Creativity-as-sensation validation.** The v0.3.6 system (valence feedback on speech creativity) has not yet been soak tested. Whether it actually reduces repetition at scale is an open question.
