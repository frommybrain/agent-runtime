# 3aiii — Status & Plan (2026-03-10)

## What Changed: v0.1 → v0.2

### Why
The v0.1 cognitive loop (OBSERVE → THINK → ACT on a fixed 8s timer) worked but was blind. The agent couldnt tell when its environment changed, didnt know if its last action succeeded or failed, had no internal states the environment could affect, and would fall into repetitive loops. These are the minimum requirements for emergent behaviour — the agent needs to *feel* its environment and *learn from consequences* without being told what to do.

Changes draw from two sources:
- Gap analysis of the existing code
- The original PROJECT_OVERVIEW cosmology design (vitality, resonance, abundance, warmth as environmental forces)

Guiding principles: pressure not timers (the loop reacts to felt activity, not a fixed schedule), constraints over instructions (the agent should be unable to repeat, not told to be creative), personality as data (structured persona file, editable and traceable, not prose embedded in a prompt), situated not sentient (the agent is shaped by its environment, not animated by an inner life).

### The New Cognitive Loop
**Before:** `OBSERVE → THINK → ACT` (fixed 8s)
**After:** `SENSE → FEEL → THINK → ACT → REFLECT` (adaptive 4-15s)

### New Modules

**InternalState** (`src/cognition/InternalState.js`)
Two-axis internal state: mood (-1..+1, bad↔good) and energy (-1..+1, calm↔excited). Sensations not instructions. Environment shifts them:
- Action results → mood (success feels good, failure feels bad)
- Environmental changes (deltas) → energy (novelty = stimulation)
- Cosmology signals → both (high resonance = energy, warmth = positive mood)
- Social events → both (being spoken to is stimulating)
- Natural decay toward neutral over time

High energy moments encoded more strongly in memory (salience multiplier).

**DeltaDetector** (`src/cognition/DeltaDetector.js`)
Diffs each observation against the previous tick. Detects:
- Agents/objects appearing or disappearing
- Own action state changes
- Available actions changing (new capability appeared)
- Environment signal shifts (resonance increased, vitality dropped)
- Arbitrary top-level data changes

Narrates changes as natural language for the LLM: "New object: terminal-01 appeared", "resonance increased (0.30 → 0.70)".

**RepetitionGuard** (`src/cognition/RepetitionGuard.js`)
Tracks last 20 actions. Detects:
- Same action 3+ times consecutively
- One action dominating >60% of recent history
- Exact same action+params repeated 3+ times in last 5

Feeds warnings to the LLM so it varies behaviour. This is the "constraints create creativity" thing, the agent has to get creative when it cant repeat.

### Modified Modules

**Perceive.js** — Env-agnostic now. No longer assumes `(x, z)` coordinates. Handles any coord system, arbitrary object properties, env signals, unknown top-level observation keys. A synth env sending `{ bpm: 120, activeChords: ["Cmaj7", "Dm9"] }` would be narrated automatically.

**WorkingMemory.js** — Salience-weighted push. High-energy moments tagged so sleep consolidation can prioritise them. Salient events marked with `*` in prompt context.

**PromptBuilder.js** — User prompt now includes (in order): TIME (tick count, uptime), INTERNAL STATE (how the agent feels), CHANGES SINCE LAST TICK (deltas), LAST ACTION RESULT (consequence feedback), NOTICE (repetition warnings), then existing sections (recent actions, today's log, current situation).

**Heartbeat.js** — Adaptive interval: high energy → faster ticking (4s, more engaged), low energy → slower ticking (15s, conserving). Uses `setTimeout` chain instead of `setInterval`. Action results captured and fed back into the next tick's THINK cycle.

**SleepCycle.js** — New Pass 4: self-reflection. The LLM reviews the agent's recent behaviour, internal state history (average mood/energy, peaks, range), and current persona. It then decides whether the agent should evolve. If yes, it proposes subtle changes to traits, quirks, values, fears, voice — never name or backstory. Changes are written to the persona JSON with a timestamped evolution log, giving the agent a record of who it used to be alongside the drift guard that anchors it to its baseline.

**ApiServer.js** — `/status` now returns `internalState` (mood, energy, labels) and `heartbeatMs` (current adaptive interval). SSE `connected` event includes internal state. Tick events include internal state, delta count, current interval.

**config.js** — New options: `heartbeatMinMs` (4000), `heartbeatMaxMs` (15000), `stateDecayRate` (0.05), `repetitionHistorySize` (20). All configurable via env vars.

**index.js** — Wires InternalState, DeltaDetector, RepetitionGuard into the Heartbeat and SleepCycle constructors. Version bumped to v0.2.

### Validation
- All 12 files pass Node.js syntax checks
- Integration test confirms: InternalState updates correctly, DeltaDetector finds changes between ticks, RepetitionGuard flags repeated actions, Perceive handles arbitrary observation shapes (incl non-spatial data like BPM and chords), WorkingMemory salience filtering works, PromptBuilder includes all new context sections
- Full app boot test passes (all imports resolve, constructors wire correctly)

## Testing Plan (2026-03-11)

### Step 1: Update the Pi

SSH onto the Pi and pull the latest code:

```bash
ssh pi@victor.local
cd ~/agent-runtime
git pull origin main
npm install  # in case any deps changed (none expected)
```

If the agent is running as a systemd service, restart it:
```bash
sudo systemctl restart agent-runtime
```

### Step 2: Run Without an Environment Server (Standalone Validation)

Before connecting to a world, verify the agent boots and the new modules initialise:

```bash
# on the Pi (or locally on Mac for faster iteration)
LOG_LEVEL=debug node src/index.js
```

Expected output:
- `=== 3aiii v0.2 ===`
- `Heartbeat: 8000ms base (adaptive 4000-15000ms)`
- Connection failure (no server). thats fine, it'll keep retrying
- No import errors or crashes

### Step 3: Create a Minimal Test Environment Server

Need a tiny WebSocket server that speaks the agent protocol. Lets us control exactly what the agent sees and validate its responses. Create this on the Mac:

```bash
# test-server.js, minimal environment server for testing
# run with: node test-server.js
```

The test server should:
1. Accept agent connection (WELCOME → IDENTIFY → IDENTIFIED)
2. Respond to OBSERVE with controllable observations
3. Accept ACT and return results (success/failure)
4. Let us inject changes between ticks (add/remove objects, change signals)

This is the key testing tool — we control the environment and watch how the agent responds.

### Step 4: Test Scenarios

Run each scenario for 5-10 minutes and observe via the API (`curl http://victor.local:5000/status`):

**Scenario A: Baseline (static world)**
- Serve the same observation every tick (no changes, no signals)
- Expected: agent explores, energy stays low, heartbeat slows toward 15s, repetition guard kicks in after a few identical actions

**Scenario B: Introduce novelty (object appears)**
- After 5 ticks, add a new interactive object to the observation
- Expected: delta detector fires "New object appeared", energy spikes, heartbeat speeds up, agent likely investigates the new object

**Scenario C: Action failure**
- When agent tries to interact with something, return `{ success: false, message: "too far away" }`
- Expected: action result feedback shows in next tick, mood dips slightly, agent tries something different

**Scenario D: Environment signals (cosmology)**
- Add `signals: { resonance: 0.8, vitality: 0.3 }` to observations
- Expected: perceiver narrates "Environment: resonance: 0.80, vitality: 0.30", energy rises from resonance, mood dips from low vitality

**Scenario E: Social event (speech)**
- Send a WORLD_EVENT with `{ event: "agent_speech", agentId: "stranger", message: "hello there" }`
- Expected: speech logged in working memory with salience, energy bumps, agent considers responding

**Scenario F: Repetition pressure**
- Serve limited actions (only `wait` and `move_to`) with no objects/agents
- Expected: after 5-6 ticks of the same action, repetition guard warns, agent varies between wait and move, eventually the warnings influence the LLM to try different patterns

### Step 5: Tracking & Logging

**Real-time monitoring:**
```bash
# watch the SSE event stream
curl -N http://victor.local:5000/events

# check current status (incl internal state, heartbeat interval)
curl http://victor.local:5000/status | jq

# view today's log
curl http://victor.local:5000/logs/today
```

**What to look for in the logs:**
- `[tick N] action (source) — reason [v=X.XX a=X.XX]` — confirms internal state is flowing
- `Delta: N change(s) detected` — confirms delta detection works
- Heartbeat interval changing in `/status` response (`heartbeatMs` field)
- Repetition warnings appearing in debug logs
- Action results showing `→ ok` or `→ failed`

**What to look for in `/status`:**
```json
{
  "internalState": {
    "mood": 0.12,
    "energy": 0.45,
    "moodLabel": "neutral",
    "energyLabel": "moderate",
    "description": "Feeling neutral with moderate energy"
  },
  "heartbeatMs": 9200
}
```

### Step 6: Validate Sleep Cycle (Accelerated)

To test self-reflection without waiting 4 hours:
```bash
# trigger sleep manually via API
curl -X POST http://victor.local:5000/sleep

# watch the logs for consolidation passes
curl http://victor.local:5000/logs/today
```

Look for:
- `Memory consolidated`
- `Skills extracted`
- `Tools refreshed`
- `Self-reflection: no evolution needed` or `Self-reflection: evolved — [reason]`
- If evolved, check the persona file for the evolution log entry

### Step 7: Non-Spatial Environment Test (Stretch Goal)

To prove env-agnosticism, modify the test server to send a completely non-spatial observation:
```json
{
  "self": { "role": "sequencer" },
  "signals": { "bpm": 120, "energy": 0.7 },
  "activeChords": ["Cmaj7", "Dm9", "G7"],
  "availableSteps": 16,
  "currentStep": 4,
  "available_actions": [
    { "name": "set_step", "params": "step, note", "description": "Set a note on a sequencer step" },
    { "name": "change_bpm", "params": "bpm", "description": "Change tempo" },
    { "name": "add_chord", "params": "chord", "description": "Add a chord to the pool" },
    { "name": "wait", "description": "Do nothing this tick" }
  ]
}
```

Expected: the perceiver narrates all of this without crashing. The LLM receives a coherent picture of a music environment. Whether it makes good decisions depends on the LLM, but the architecture handles it.
